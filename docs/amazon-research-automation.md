## 継続実行方式

`researchListedItemsHourly()` は実行冒頭でメイン表からリサーチ管理表へ注文番号同期を行い、その後はリサーチ管理表で非表示になっていない行を収集します。`候補あり`や`候補なし`を含め、リサーチ状況は対象除外に使いません。列位置は固定せず、既存ヘッダーから判定します。

Apps Scriptの実行時間上限に備え、処理時間または行数の上限へ到達した場合はスクリプトプロパティにカーソルを保存します。次の1時間トリガーはカーソル位置から再開するため、先頭行だけが繰り返し処理されることを防ぎます。

各実行の冒頭で`syncResearchManagementByOrderNumber()`相当の同期を行います。同期の基準は常に`注文確定商品リサーチ表`です。`注文確定商品リサーチ表`に表示されている注文番号は`リサーチ管理表`にも表示し、`注文確定商品リサーチ表`から注文自体が削除された場合のみ管理表の対応行を削除します。`リサーチ管理表`側の有無・候補URL・リサーチ結果・ステータスを基準に、`注文確定商品リサーチ表`へ注文を追加または削除することはありません。

Gmail自動取得で新規注文行が追加された場合は、追加した注文番号リストを使って`リサーチ管理表`へ即時同期し、その注文だけを対象にリサーチを開始します。これにより、注文確定商品リサーチ表への自動取得から管理表側のリサーチ開始までが同一実行内で連動します。既存の表示行全体は再処理しないため、手入力・既存URL・手動非表示行は従来どおり保護されます。

リサーチ管理表は自動リサーチの実行対象シートであり、削除しません。`削除済み注文`はユーザーが削除した注文番号の再表示を防ぐために使用するため、こちらも削除しません。

候補URLの追記は、単にURLが新しいだけでは行いません。既存候補と比較して、合計価格がより安い、または状態ランクがより良い場合だけ追記します。価格・状態のどちらも改善しない候補は確認用にも追加せず、既存の良い候補を優先します。

注文メール取り込み前と行変更トリガー実行時には、注文番号スナップショットとの差分を確認します。スナップショットに存在して現在のメイン表に存在しない注文番号は`削除済み注文`へ記録し、以後のGmail取り込みでは再表示しません。132行目以降をまとめて削除する専用メニューも同じ記録処理を行います。

さらに、セットアップ・Gmail取り込み・行変更トリガーの入口で132行目以降に残っている注文行を自動的に`削除済み注文`へ登録して削除します。これにより、ユーザーがメニューを実行できない場合でも過去に削除した領域を再表示しません。

## 2026-07-02 追記: 管理表限定追記と旧シート削除

- 候補URLの追記は `リサーチ管理表` 側だけに行います。`注文確定商品リサーチ表` 側のF〜K列へは追記しません。
- 新しい候補URLの追記先はF〜K列（Amazon / ヤフオク / メルカリ / ジモティ / 楽天市場 / その他サイト）です。
- `リサーチ管理シート` は旧名称の未使用シートとして扱い、セットアップ時・管理表同期時・専用メニュー実行時に存在すれば削除します。
- `リサーチ管理表` は自動リサーチの対象、`削除済み注文` は削除済み注文の再表示防止に必要なため、どちらも削除しません。
- 既存の `リサーチ管理表` でも、F〜K列ヘッダーは `Amazon` / `ヤフオク` / `メルカリ` / `ジモティ` / `楽天市場` / `その他サイト` に補正します。K列が旧メタ列の場合はL/Mへ退避し、既存URL・メモ・手入力・色・書式は保持します。
- 自動トリガーを待たずに表示行を処理するため、`researchAllVisibleManagementRowsNow` を追加します。既存メニューの `リサーチを手動実行` もこの即時リサーチへ接続します。
- Gmail自動取得直後の管理表追加と対象注文のみの即時リサーチ
- 削除済み注文番号の記録とGmail再追加防止
- リサーチ管理表表示行の毎時リサーチ
- リサーチ管理表J列への楽天市場候補追記
- リサーチ管理表K列へのその他サイト候補追記
