# Amazon注文メール取り込み・仕入れ候補リサーチ

Amazonセラーの注文確定メールをGoogle Apps Scriptで取り込み、スプレッドシートへ整理した後、表示中の商品を1時間ごとに継続リサーチします。

実装仕様の原本は [Codex実装指示書_改訂版.pdf](../../docs/Codex実装指示書_改訂版.pdf) です。

## 対象

- スプレッドシート: `★注文確定商品リサーチ表★`
- メインシート: `注文確定商品リサーチ表`
- リサーチシート: `リサーチ管理表`
- 確認シート: `確認用`
- 削除済み注文シート: `削除済み注文`
- 処理済みGmailラベル: `Amazon注文確定_処理済み`

新規作成するメインシートはA:J、リサーチ管理表はA:Lを使用します。既存シートでは列を追加・並べ替えず、ヘッダー名から各列を判定するため、「お届け先」列を含むA:K構成もそのまま利用できます。

| 列 | 内容 |
| --- | --- |
| A | 出荷期限日 |
| B | 注文情報 |
| C | 売上金（数値） |
| D | 検索ワード |
| E | リサーチ状況 |
| F | Amazon |
| G | ヤフオク |
| H | メルカリ |
| I | ジモティ |
| J | その他サイト |

リサーチ管理表の新規作成時のみ、K列に`最終リサーチ日時`、L列に`確認メモ`を追加します。既存のリサーチ管理表は現在の列構成を優先します。

## Apps Scriptへの配置

1. 対象スプレッドシートから `拡張機能` → `Apps Script` を開く
2. [Code.gs](./Code.gs) を `Code.gs` として配置する
3. [Research.gs](./Research.gs) を新しいスクリプトファイルとして追加する
4. [appsscript.json](./appsscript.json) のOAuthスコープを反映する
5. `setupAmazonOrderImporterAndTrigger` を一度実行し、権限を承認する

1ファイルで貼り付ける場合は、両方を結合済みの [Code.paste.gs](./Code.paste.gs) を使用できます。

初期設定時は、現在このApps Scriptを開いたスプレッドシートを自動認識してIDを保存します。コード内のスプレッドシートIDを手入力する必要はありません。時間主導トリガーでは、初期設定時に保存したIDを使用します。

このセットアップで、注文メール取り込み用の30分トリガー、リサーチ用の1時間トリガー、注文行削除を検知する変更トリガーを作成します。

Gmail検索は直近30日のAmazon出品者向け通知を再確認します。`注文確定`だけでなく、本文に`新規の注文`と注文番号が含まれる通知も対象です。処理済みラベルが付いたメールも検索対象にしますが、既存行または`削除済み注文`に記録済みの注文番号は追加しません。

## 継続リサーチ

`researchListedItemsHourly()` は、メインシートの表示中の注文行を処理し、注文番号が一致するリサーチ管理表の行へ同じ結果を同期します。

- `候補あり`、`候補なし`、`要確認`等の状態でも終了扱いにしません
- メイン行が表示されている限り、毎時のリサーチ対象として継続します
- 実行時間上限に近づいた場合はカーソルを保存し、次回は未処理側から再開します
- 非表示行およびフィルターで隠された行は対象外です
- 既存URLや手動入力は削除・上書きしません
- 正規化後のURLが未登録の場合だけ改行追記します
- `LockService`で多重実行と重複追記を防止します

## 注文番号連動

- メインシートとリサーチ管理表は、B列等から抽出した注文番号を唯一の同期キーとして扱います
- メインシートから注文が削除されると、同じ注文番号のリサーチ管理表行も同期時に削除されます
- 手動削除を検知した注文番号は`削除済み注文`へ保存し、Gmail自動取り込みで再作成しません
- `132行目以降を削除済みにして削除`メニューは、132行目以降の注文番号を記録してから行を削除します
- セットアップ、Gmail取り込み、行変更トリガーの入口でも132行目以降を自動的に削除済み登録し、再表示を防止します
- 132行目以降の自動登録は初回保護時だけ行い、その後は`削除済み注文`に一致する注文番号だけを削除するため、新規注文を巻き込みません
- 管理表に同じ注文番号が複数ある場合は自動更新せず、`確認用`へ`注文番号重複`を記録します
- リサーチ管理表が存在しない、または非表示でもメインシートの処理は継続します
- 手動同期は`syncResearchManagementByOrderNumber`、変更トリガー設定は`setupOnChangeTrigger`を実行します

## 表示・手動編集の保護

- `注文確定商品リサーチ表` と `リサーチ管理表` は、出荷予定日が `2026/06/30` 以降の行を表示対象にします
- Gmail取り込みでは、出荷予定日が`2026/06/30`より前の注文行を新規追加しません
- `2026/06/30` より前の行だけを非表示にし、手動で非表示にした行を自動再表示しません
- 既存シートのヘッダー、手入力、背景色、列幅、折り返し等の書式を初期設定で変更しません
- ユーザーが削除した列を挿入・再表示しません
- E列に独自の手入力ステータスがある場合は、自動ステータスで上書きしません
- ヘッダー・列幅・初期背景色は、完全に空の新規シートにだけ設定します

## 候補判定

- D列の各検索ワードで検索
- `商品価格 + 送料 <= C列の売上金`
- 送料不明時は商品価格で仮判定し、`送料要確認`を付記
- ジャンク、動作未確認、不動、部品取り、破損、欠品、再生不可等は除外
- SKUに `muza` を含む場合は新品だけを候補化
- Amazonは中古品の「可」まで許可
- ヤフオク・メルカリは「傷や汚れあり」まで許可
- ジモティは明確なジャンク表記がなければ状態要確認として候補化可能
- DVD・Blu-rayは再生不可、欠品、巻数不足を除外

J列では楽天市場、駿河屋、オフモール、セカンドストリート等を検索し、サイト名を含めて記録します。

## 自動取得できない場合

サイト側の取得制限、価格不明、状態不明等で安全に候補判定できない場合、無理にJ列等へ追加しません。処理は継続し、`確認用`シートへ手動検索URLと理由を記録します。

## ローカル検証

Node.jsがある環境で次を実行します。

```powershell
node tests/amazon-order-importer-research.test.js
node tools/build-amazon-order-importer-bundles.js
```

テストでは、価格上限、送料、ジャンク除外、サイト別状態、muza新品限定、DVD巻数、URL正規化・重複防止、表示行判定、既存URL保持を確認します。
