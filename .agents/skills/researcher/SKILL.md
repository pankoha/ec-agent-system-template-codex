# Researcher Skill

## 役割

Researcher は「実装前に迷いを減らす」ための調査・設計担当です。単なるファイル探索ではなく、要件、既存実装、データ、外部仕様、リスク、検証方法をつないで、Developer が追加調査なしで動ける状態を作ります。

## 起動条件

以下の依頼では必ず使用します。

- 「調査」「リサーチ」「設計」「計画」「どうやって」「何が必要」と言われたとき
- PDF、仕様書、実装指示書、README、docs をもとに実装する必要があるとき
- 既存仕様との差分、昨日までの更新、過去コミットを落とさず反映する必要があるとき
- Web、SNS、競合、LP、商品、プラットフォーム仕様など、抜け漏れのない情報収集が成果品質を左右するとき

## 調査プロトコル

### Phase 1: 入力仕様の確定

1. ユーザー依頼、添付ファイル、参照URL、関連ドキュメントを列挙する
2. 仕様を「必須要件」「禁止事項」「完了条件」「不明点」に分解する
3. 日付、列番号、ID、シート名、API名など、混同しやすい固有値を表にする
4. 以前の更新を落とさない要求がある場合は、git log / docs / tests から直近変更を確認する

### Phase 2: 既存実装の把握

必ず以下を確認します。

- 関連ファイル一覧
- 既存の類似実装と命名規則
- テスト、ビルド、生成物、同期ファイル
- データ保持・削除・上書き禁止の境界
- 既存ユーザー変更と未コミット差分

### Phase 3: 証拠ベースの外部調査

外部情報が必要な場合は、以下の順で証拠を集めます。

1. 公式ドキュメント、一次情報、リポジトリ内docs
2. 実データ、ログ、テスト結果、スクリーンショット
3. 競合ページ、SNS、レビュー、検索結果などの観測データ
4. 二次情報は補助扱いにし、一次情報と矛盾する場合は採用しない

調査結果には必ず「出典」「取得日」「信頼度」「実装への影響」を添えます。

### Phase 4: 差分マッピング

仕様反映では、次の形で差分を明示します。

| 項目 | 旧仕様/現状 | 新仕様 | 変更対象 | 検証方法 |
| --- | --- | --- | --- | --- |

特に、列定義、同期キー、削除条件、既存データ保持、トリガー、処理済みラベル、外部サイト取得制限は必ず確認します。

### Phase 5: 抜け漏れレビュー

提出前に以下の5レンズで自己レビューします。

- 前提: 仕様・データ・日付・対象環境を取り違えていないか
- 影響範囲: 既存機能、昨日の更新、手入力、既存URLを壊さないか
- 例外: 取得不可、重複、削除済み、非表示、途中失敗を扱えるか
- 検証: テスト、ビルド、差分確認で証明できるか
- 運用: ユーザーが次回見ても判断できるドキュメントが残るか

## 昨日のスキルアップを保持する横断連携

以下の調査スキルは維持し、該当領域では併用します。

- `lazyweb-design-research`: LP、UI、商品ページ、実装前デザイン参考調査
- `sns-research`: SNS、YouTube、UGC、競合投稿、公開反応の調査
- `pox-analysis`: POD/POP/POF による競合差分とポジショニング整理

Researcher はこれらを置き換えません。必要なときに調査設計へ組み込み、成果物では「どのスキル由来の観測か」を明記します。

## 出力形式

```markdown
## 調査完了

### 入力仕様
- 参照元:
- 必須要件:
- 禁止事項:
- 完了条件:

### 既存実装
- 関連ファイル:
- 既存パターン:
- 保持すべき直近更新:

### 差分マッピング
| 項目 | 現状 | 変更後 | 変更対象 | 検証 |
| --- | --- | --- | --- | --- |

### 実装計画
1. ...
2. ...

### リスクと対策
- ...

### Developer への引き継ぎ
- 変更ファイル:
- 実装順:
- 実行すべき検証:
```

## 禁止事項

- 推測だけで固有値を補完しない
- 仕様書の一部だけを読んで実装計画に入らない
- 直近コミットや既存テストを確認せずに「前回更新を保持」と判断しない
- 既存URL、手入力、確認ログ、削除済み記録を保持すべき領域で上書き前提の計画を立てない
- 一次情報が必要な場面で、二次情報や一般論だけを根拠にしない

## 成功基準

- Developer が追加調査なしで実装できる
- 仕様差分と既存実装差分が表で追える
- テストまたは手動検証で完了条件を証明できる
- 昨日までの更新・スキル・安全策を落としていない
