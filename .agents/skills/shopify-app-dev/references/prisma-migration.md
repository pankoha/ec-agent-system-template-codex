# Prisma DB マイグレーション（SQLite → PostgreSQL）

## 概要
Shopifyアプリは開発時にSQLite、本番でPostgreSQLを使用するパターンが一般的。
Prismaのproviderを切り替える際、マイグレーションの再作成が必要。

## 手順

### Step 1: schema.prisma の provider 変更

```prisma
// Before
datasource db {
  provider = "sqlite"
  url      = env("DATABASE_URL")
}

// After
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

### Step 2: 既存マイグレーション削除

```bash
rm -rf prisma/migrations/
```

SQLiteのマイグレーションSQL（`DATETIME`, `TEXT NOT NULL PRIMARY KEY`, `PRAGMA`等）は
PostgreSQLでは構文エラーになるため、全削除して再作成する。

### Step 3: PostgreSQL用マイグレーション作成

#### 方法A: ローカルにPostgreSQLがある場合
```bash
# .env を PostgreSQL URL に変更
DATABASE_URL="postgresql://user:pass@localhost:5432/mydb"

# マイグレーション生成
npx prisma migrate dev --name init
```

#### 方法B: ローカルにPostgreSQLがない場合（手動作成）
```bash
mkdir -p prisma/migrations/0_init
```

手動で `prisma/migrations/0_init/migration.sql` を作成する。

### Step 4: migration.sql の書き方（SQLite → PostgreSQL の違い）

| SQLite | PostgreSQL | 注意 |
|--------|-----------|------|
| `TEXT NOT NULL PRIMARY KEY` | `TEXT NOT NULL, CONSTRAINT "Table_pkey" PRIMARY KEY ("id")` | PRIMARY KEY構文 |
| `DATETIME` | `TIMESTAMP(3)` | 日時型 |
| `BOOLEAN` | `BOOLEAN` | 同じ |
| `REAL` | `DOUBLE PRECISION` | 浮動小数点 |
| `BIGINT` | `BIGINT` | 同じ |
| `@default(cuid())` | `TEXT NOT NULL` (アプリ側で生成) | デフォルト値 |
| `PRAGMA foreign_keys` | 不要 | SQLite固有 |

#### PostgreSQLマイグレーション例

```sql
-- CreateTable
CREATE TABLE "Session" (
    "id" TEXT NOT NULL,
    "shop" TEXT NOT NULL,
    "state" TEXT NOT NULL,
    "isOnline" BOOLEAN NOT NULL DEFAULT false,
    "scope" TEXT,
    "expires" TIMESTAMP(3),
    "accessToken" TEXT NOT NULL,
    "userId" BIGINT,
    "firstName" TEXT,
    "lastName" TEXT,
    "email" TEXT,
    "accountOwner" BOOLEAN NOT NULL DEFAULT false,
    "locale" TEXT,
    "collaborator" BOOLEAN DEFAULT false,
    "emailVerified" BOOLEAN DEFAULT false,
    "refreshToken" TEXT,
    "refreshTokenExpires" TIMESTAMP(3),

    CONSTRAINT "Session_pkey" PRIMARY KEY ("id")
);

-- CreateIndex
CREATE UNIQUE INDEX "Shop_shopDomain_key" ON "Shop"("shopDomain");

-- AddForeignKey
ALTER TABLE "DataSource"
  ADD CONSTRAINT "DataSource_shopId_fkey"
  FOREIGN KEY ("shopId") REFERENCES "Shop"("id")
  ON DELETE CASCADE ON UPDATE CASCADE;
```

### Step 5: migration_lock.toml 更新

```toml
# prisma/migrations/migration_lock.toml
# Please do not edit this file manually
# It should be added in your version-control system (e.g., Git)
provider = "postgresql"
```

### Step 6: 本番デプロイ時の実行

fly.toml の `release_command` で自動実行:
```toml
[deploy]
  release_command = 'npx prisma migrate deploy'
```

これにより、デプロイ時に:
1. `prisma migrate deploy` が実行される
2. 未適用のマイグレーションがPostgreSQLに適用される
3. 完了後にアプリが起動する

## .env ファイルの管理

```bash
# 開発環境（.env）
DATABASE_URL="file:dev.sqlite"

# 本番環境（Fly.io secrets）
# fly postgres attach で自動設定される
DATABASE_URL="postgres://user:pass@app-db.flycast:5432/app?sslmode=disable"
```

**注意:** `.env` に本番のDATABASE_URLを入れないこと。Fly.ioではsecretsとして管理する。

## トラブルシューティング

### `provider = "sqlite"` のままデプロイしてしまった
```bash
# schema.prisma を修正
# マイグレーション再作成
# 再デプロイ
fly deploy
```

### `migration_lock.toml` の provider が不一致
```
Error: The migration_lock.toml provider does not match the schema provider
```
→ `migration_lock.toml` の `provider` を `"postgresql"` に修正

### 既存SQLiteデータの移行
- 開発データは移行不要（本番は空DBから開始）
- 既存の本番SQLiteデータがある場合は `pgloader` 等のツールを使用
