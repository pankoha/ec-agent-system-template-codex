# スコープ定義 & WBS テンプレート

## スコープ定義
```
【やること（In Scope）】
1.
2.
3.

【やらないこと（Out of Scope）】
1.
2.
3.

【前提条件】
1.
2.

【制約条件】
1.
2.
```

## WBS（Work Breakdown Structure）
```
WBSの本質はスケジューリングではなく、
「依存関係」と「抜け漏れ」の発見にある。

MECE（Mutually Exclusive, Collectively Exhaustive）であること。
```

### WBSテンプレート
```markdown
| Task ID | Task Name | Dependency | Owner | Start | End | Status |
|---------|-----------|------------|-------|-------|-----|--------|
| T-001 | 要件定義 | - | 担当A | 05/01 | 05/10 | Done |
| T-002 | デザイン制作 | T-001 | 担当B | 05/13 | 05/24 | In Progress |
| T-003 | コーディング | T-002 | 担当C | 05/27 | 06/07 | Pending |
| T-004 | テスト | T-003 | 担当D | 06/10 | 06/14 | Pending |
| T-005 | リリース | T-004 | 担当A | 06/17 | 06/17 | Pending |
```

## クリティカルパス分析
```
依存関係の連鎖で最長となるパスを特定
→ ここが遅れると全体が遅れる
→ 重点的にリソースを配分
```

## タスクステータス定義
```
🔵 Not Started: 未着手
🟡 In Progress: 進行中
🟢 Done: 完了
🔴 Blocked: ブロック中（依存関係で停止）
⚫ Cancelled: 取消
```

## ルーティン管理表テンプレート
```markdown
| 頻度 | タスク | 担当 | 実行時間 | チェック方法 | IF-THEN |
|------|--------|------|---------|-------------|---------|
| 日次 | 競合監視 | A | 9:00 | Slack報告 | 変更あれば即共有 |
| 週次 | KPI集計 | B | 月曜10:00 | スプシ更新 | 異常値は即報告 |
| 月次 | 請求書発行 | C | 25日 | 発行完了報告 | 未入金は翌月2日催促 |
```

## 標準化された指示書（IF-THEN ルール）
```
アシスタントが「脳を使わずに」実行できるレベルまで標準化する。

IF: ランキングが3位以上下落
THEN: ディレクターにアラート

IF: 競合がオファー変更
THEN: スクリーンショットしてSlackに投稿

IF: 在庫が7日分以下
THEN: 発注依頼を起票
```
