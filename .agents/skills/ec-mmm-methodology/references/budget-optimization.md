# 予算最適化の実践ガイド

## 最適化アルゴリズムの仕組み

### 限界ROI均衡化（Marginal ROI Equalization）

```
目的: 全広告チャネルの限界ROIが等しくなるように予算を配分する
原理: 限界ROIの高いチャネルに予算を移動 → 限界ROIが均一化 → 全体効率が最大化

計算ステップ:
Step 1: 各チャネルのHill飽和曲線パラメータ（K, S）を取得
Step 2: 現在の支出で限界ROI（Hill関数の微分値）を算出
Step 3: 限界ROIが高いチャネルに予算を移動（勾配ステップ法、50回反復）
Step 4: 上限・下限制約を適用
Step 5: 総予算制約に合わせて正規化

Hill関数の微分（限界リターン）:
marginal = S × K^S × spend^(S-1) / (K^S + spend^S)^2

意味:
├── 支出が少ない → 限界リターンが大きい（投資余地あり）
├── 支出が多い → 限界リターンが小さい（飽和に近い）
└── K付近が最も限界リターンの変化が大きい（半飽和点）
```

### 制約条件

```
上限制約（max_multiplier = 3.0）:
├── 既存チャネル: 現在支出 × 3.0 まで
├── 新規チャネル（支出ゼロ）: 総予算 / チャネル数 まで
└── 理由: 急激な増額は実行リスクが高い。段階的に増額するための安全弁

下限制約（min_ratio = 0.1）:
├── 既存チャネル: 現在支出 × 0.1 まで（最低10%は維持）
├── 新規チャネル: 0（ゼロまで削減可能）
└── 理由: 完全停止はベースライン効果を失う。最低限の存在感を維持

プラットフォームチャネル:
├── 最適化対象外（手数料ベースで直接コントロール不可）
├── Shopify, BASE, MakeShop, Yahoo Stores, WooCommerce 等
└── 売上構成比・コスト構造の分析対象には含めるが、予算配分からは除外
```

### キャップ（capped）の意味

```
capped = "" (空文字):
├── 上限・下限のどちらにも引っかかっていない
├── 限界ROIが他チャネルと均衡した適正配分
└── → この推奨値は高い信頼度で実行可能

capped = "upper":
├── 上限制約（3.0倍）に達している
├── 制約がなければもっと増やすべき（限界ROIがまだ高い）
├── → 段階的に増額し、次回MMM実行で上限を緩和して再分析
└── → 実行後に限界ROIの変化をモニタリング

capped = "lower":
├── 下限制約（0.1倍）に達している
├── 制約がなければもっと減らすべき（限界ROIが極めて低い）
├── → このチャネルの存在意義を再検討
├── → ブランド認知目的なら維持。それ以外は本当に必要か精査
└── → 完全停止テスト（2-4週間）で影響を計測するのも選択肢
```

## 推奨予算の解釈と実行

### 変動幅別アクションプラン

```
推奨変動: +200%（上限キャップ）
├── 意味: 投資余地が非常に大きい。限界ROIがまだ高い
├── リスク: 急激な増額はCPAスパイクを招く可能性
├── アクション:
│   ├── 1週目: +50%で開始
│   ├── 2週目: CPAとCVRをモニタリング
│   ├── 3週目: 問題なければ+100%に増額
│   └── 4週目: 再度MMM実行で効果検証
└── 注意: オーディエンスサイズ・在庫キャパの確認が必要

推奨変動: +50%〜100%
├── 意味: 明確な増額推奨。限界ROIが平均より高い
├── リスク: 中程度。クリエイティブ・LP準備が必要
├── アクション:
│   ├── 2週間で段階的に目標額まで増額
│   ├── 週次でROAS・CPA変化をチェック
│   └── 月次MMM再実行で飽和度変化を確認
└── 注意: 競合の動向で急に効率が変わる場合あり

推奨変動: -10%〜+10%
├── 意味: ほぼ適正配分。大きな変更は不要
├── アクション: 現状維持。他のチャネルに注力
└── 注意: 微調整よりもクリエイティブ最適化の方が効果的

推奨変動: -10%〜-30%
├── 意味: 軽度の過剰投資。限界ROIが平均以下
├── アクション:
│   ├── 2-3週間で段階的に削減
│   ├── 削減分を限界ROIの高いチャネルに再配分
│   └── 売上への影響をモニタリング（シナジーに注意）
└── 注意: 相関の高いチャネルへの波及効果を確認

推奨変動: 下限キャップ（-90%）
├── 意味: 限界ROIが極めて低い。過飽和状態
├── アクション:
│   ├── まずは-30%削減テスト（2週間）
│   ├── 売上影響が限定的なら追加-30%
│   ├── 削減しても影響がなければ最小維持レベルまで
│   └── MMM再実行で飽和度の改善を確認
└── 注意: 完全停止は避ける。ベースライン効果の喪失リスク
```

### 限界ROIの具体的な読み方

```
限界ROI（marginal_roi）の判断基準:

10.0+ → 投資不足が深刻。黄金ゾーン。最優先で増額
├── 例: 楽天 marginal_roi = 10.26 → 1円追加で10.26円のリターン
└── アクション: 可能な限り急速に増額。在庫・オペ体制を確認

2.0-10.0 → 十分な投資余地あり。積極増額推奨
├── 例: Meta Ads marginal_roi = 2.55 → まだ効率的に伸ばせる
└── アクション: 月次20-50%増額ペースで段階的に

1.0-2.0 → 損益分岐点付近。慎重な増額 or 現状維持
├── ROI > 粗利率なら増額余地あり（ROI=1.5でも粗利率65%なら利益）
└── アクション: 月次5-10%の微増。限界ROI < 1.0になったら停止

0.5-1.0 → 追加投資は赤字。削減検討
├── ただし粗利率が高い商材ならまだ許容範囲の場合も
└── アクション: 段階的に-10%ずつ削減。他チャネルに再配分

0.0 → 完全飽和。追加投資は100%無駄
├── Hill R²が低い場合、この数値自体が信頼できない可能性
└── アクション: まずHill R²を確認。低ければデータ品質の改善を優先
```

### 限界CPA（marginal_cpa）の活用

```
限界CPA = 1円追加投資した時の追加1件あたりの獲得コスト

計算式:
marginal_cpa = 1.0 / (marginal_roi × orders_per_revenue)
※ orders_per_revenue = 期間注文数 / 期間売上

判断基準:
├── 限界CPA < 平均CPA → まだ効率的に獲得可能。増額余地あり
├── 限界CPA ≈ 平均CPA → 適正水準。現状維持
├── 限界CPA > 平均CPA × 1.5 → 効率が急落。増額停止
├── 限界CPA > LTV → 赤字獲得。削減必須
└── 限界CPA = ∞ (Infinity) → 追加投資で獲得不可。完全飽和

True CPA vs 限界CPA:
├── True CPA: アドストック考慮コスト / 注文数（残存効果を含む真のCPA）
├── 限界CPA: 次の1件を追加獲得するコスト
└── True CPA < 限界CPA が通常（過去蓄積 vs 追加分のコスト差）

活用シナリオ:
├── 「限界CPAが¥15,000を超えたのでMeta Adsの増額を停止」
├── 「限界CPAが¥1,500の楽天に予算を集中投下」
└── 「LTV ¥12,000に対し限界CPA ¥14,000 → このチャネルは赤字獲得」
```

## Budget Preview（予算シミュレーション）の活用

```
2つのモード:

1. Auto-optimize（自動最適化）:
   └── total_budget のみ指定 → アルゴリズムが最適配分を計算
   └── 全体予算の変更（増減）シミュレーションに使用
   └── 例: 「総予算を20%増やしたらどうなるか？」

2. Manual scenario（手動シナリオ）:
   └── channel_budgets で各チャネルの予算を個別指定
   └── 特定チャネルの増減テストに使用
   └── 例: 「Meta Adsを-20%、楽天を+100%にしたらどうなるか？」

活用パターン:
├── 現状予算 → Auto-optimize → 最適配分との差を確認
├── 最適配分 → Manual で段階的に近づける → 実行可能性を確認
├── Best/Base/Worst 3シナリオ → 予算±20%で比較
└── チャネル追加/削除テスト → 1チャネルをゼロにした影響を確認
```

## 予算最適化のトラブルシューティング

```
症状: 全チャネルが上限キャップ（+200%）
├── 原因: 総予算が現在支出より大幅に多い
└── 対処: 総予算を現在支出と合わせて再実行

症状: 1チャネルに予算が偏る
├── 原因: そのチャネルのHill K値が著しく大きい
└── 対処: Kパラメータの妥当性を確認。データ不足ならフォールバック値を使用

症状: marginal_roi が全チャネル 0.0
├── 原因: 全チャネルが過飽和（Hill曲線の天井に到達）
└── 対処: Kパラメータの推定失敗の可能性。データ期間の延長を検討

症状: 推奨配分が直感と大きく異なる
├── 原因1: アドストック減衰率が不適切（デフォルト値が実態に合わない）
├── 原因2: データ期間にセール等の異常値が含まれている
├── 原因3: チャネル間の多重共線性が高い
└── 対処: 減衰率を調整して再実行。外部要因を登録。相関行列を確認

症状: 最適化前後で予測売上が変わらない
├── 原因: 全チャネルの飽和度が高く、再配分の余地が少ない
└── 対処: 新規チャネルの追加を検討。または総予算の適正化（削減含む）
```

## 実行プランニングテンプレート

```
予算変更の実行チェックリスト:

□ Phase 1: 分析確認（Day 1）
  ├── MMM結果のR²/MAPEを確認（十分な精度か？）
  ├── 飽和度とHill R²の整合性を確認
  ├── シナジーマップでチャネル間の依存関係を把握
  └── 推奨予算の変動幅と制約（capped）を確認

□ Phase 2: 影響評価（Day 2-3）
  ├── 増額チャネル: オーディエンスサイズ・在庫は十分か？
  ├── 削減チャネル: シナジーのある他チャネルへの影響は？
  ├── Budget Preview で3シナリオ（Best/Base/Worst）を試算
  └── CFO/OPERATIONSと連携して実行可能性を確認

□ Phase 3: 段階実行（Week 1-2）
  ├── 推奨変動の50%を初週で実行
  ├── 日次でROAS/CPA/CVRをモニタリング
  ├── 異常値があれば即座にロールバック
  └── 問題なければ2週目で残り50%を実行

□ Phase 4: 効果検証（Week 3-4）
  ├── 新しいデータでMMM再実行
  ├── 飽和度・限界ROIの変化を確認
  ├── 予測と実績の乖離を分析
  └── 次回の予算配分に反映
```
