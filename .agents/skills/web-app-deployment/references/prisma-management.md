# Prismaデータベース管理 — 汎用Webアプリ

## 概要
Prisma ORMを使用するWebアプリの開発→本番データベース管理手順。
開発環境（SQLite）→ 本番環境（PostgreSQL）の移行パターンを含む。

## 開発環境 vs 本番環境

| 項目 | 開発環境 | 本番環境 |
|------|---------|---------|
| DB | SQLite (`file:dev.sqlite`) | PostgreSQL |
| マイグレーション | `prisma migrate dev` | `prisma migrate deploy` |
| データ | テストデータ | 本番データ（保護対象） |
| 接続先 | ローカルファイル | Fly.io PostgreSQL / 外部DB |
| スキーマ変更 | 自由に変更可 | 慎重に（ダウンタイム考慮） |

## SQLite → PostgreSQL マイグレーション

### Step 1: schema.prisma の provider 変更

```prisma
// Before（開発用）
datasource db {
  provider = "sqlite"
  url      = env("DATABASE_URL")
}

// After（本番用）
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

### Step 2: 既存SQLiteマイグレーション削除

```bash
rm -rf prisma/migrations/
```

SQLiteのマイグレーションSQL（`DATETIME`, `TEXT NOT NULL PRIMARY KEY`, `PRAGMA`等）は
PostgreSQLでは構文エラーになるため、全削除して再作成する。

### Step 3: PostgreSQL用マイグレーション作成

#### 方法A: ローカルにPostgreSQLがある場合
```bash
# .env を PostgreSQL URL に変更
DATABASE_URL="postgresql://user:pass@localhost:5432/mydb"

# マイグレーション生成
npx prisma migrate dev --name init
```

#### 方法B: ローカルにPostgreSQLがない場合（手動作成）
```bash
mkdir -p prisma/migrations/0_init
```

手動で `prisma/migrations/0_init/migration.sql` を作成する（Step 4参照）。

### Step 4: SQLite → PostgreSQL 型マッピング

| SQLite | PostgreSQL | 注意 |
|--------|-----------|------|
| `TEXT NOT NULL PRIMARY KEY` | `TEXT NOT NULL, CONSTRAINT "Table_pkey" PRIMARY KEY ("id")` | PRIMARY KEY構文 |
| `DATETIME` | `TIMESTAMP(3)` | 日時型 |
| `BOOLEAN` | `BOOLEAN` | 同じ |
| `REAL` | `DOUBLE PRECISION` | 浮動小数点 |
| `BIGINT` | `BIGINT` | 同じ |
| `INTEGER` | `INTEGER` | 同じ |
| `@default(autoincrement())` | `SERIAL` or `GENERATED ALWAYS AS IDENTITY` | 自動採番 |
| `@default(cuid())` | `TEXT NOT NULL` (アプリ側で生成) | デフォルト値 |
| `PRAGMA foreign_keys` | 不要 | SQLite固有 |

#### PostgreSQLマイグレーション例

```sql
-- CreateTable
CREATE TABLE "User" (
    "id" TEXT NOT NULL,
    "email" TEXT NOT NULL,
    "name" TEXT,
    "createdAt" TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
    "updatedAt" TIMESTAMP(3) NOT NULL,

    CONSTRAINT "User_pkey" PRIMARY KEY ("id")
);

-- CreateTable
CREATE TABLE "Post" (
    "id" TEXT NOT NULL,
    "title" TEXT NOT NULL,
    "content" TEXT,
    "published" BOOLEAN NOT NULL DEFAULT false,
    "authorId" TEXT NOT NULL,
    "createdAt" TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT "Post_pkey" PRIMARY KEY ("id")
);

-- CreateIndex
CREATE UNIQUE INDEX "User_email_key" ON "User"("email");

-- AddForeignKey
ALTER TABLE "Post"
  ADD CONSTRAINT "Post_authorId_fkey"
  FOREIGN KEY ("authorId") REFERENCES "User"("id")
  ON DELETE CASCADE ON UPDATE CASCADE;
```

### Step 5: migration_lock.toml 更新

```toml
# prisma/migrations/migration_lock.toml
# Please do not edit this file manually
# It should be added in your version-control system (e.g., Git)
provider = "postgresql"
```

### Step 6: 本番デプロイ時の実行

fly.toml の `release_command` で自動実行:
```toml
[deploy]
  release_command = 'npx prisma migrate deploy'
```

デプロイ時の流れ:
1. Dockerイメージがビルドされる
2. `release_command` が実行される（`prisma migrate deploy`）
3. 未適用のマイグレーションがPostgreSQLに適用される
4. 完了後にアプリが起動する

## 日常のスキーマ変更フロー

### 開発環境での変更

```bash
# 1. schema.prisma を編集
# 2. マイグレーション生成（開発DB自動更新）
npx prisma migrate dev --name add_user_avatar

# 3. Prisma Client再生成
npx prisma generate
```

### 本番環境への反映

```bash
# fly deploy で自動的に prisma migrate deploy が実行される
fly deploy
```

**注意:** `prisma migrate dev` は開発専用。本番では必ず `prisma migrate deploy` を使用。

## Prisma Client生成

```bash
# Prisma Client生成（schema.prismaからTypeScript型を生成）
npx prisma generate
```

**Dockerfileでの配置:**
```dockerfile
COPY . .
RUN npx prisma generate  # ← ビルド前に実行
RUN npm run build
```

`prisma generate` はビルド時に実行する。`node_modules/.prisma/client/` にクライアントが生成される。

## .env ファイル管理

```bash
# 開発環境（.env）— gitignore対象
DATABASE_URL="file:dev.sqlite"

# 本番環境（Fly.io secrets）— .envには書かない
# fly postgres attach で自動設定される
DATABASE_URL="postgres://user:pass@app-db.flycast:5432/app?sslmode=disable"
```

**絶対にやってはいけないこと:**
- `.env` に本番のDATABASE_URLを書く
- DATABASE_URLをGitにコミットする
- 本番DBに `prisma migrate dev` を実行する

## Prisma Studio（開発用GUIツール）

```bash
# ローカルのDBをブラウザで閲覧・編集
npx prisma studio
```

**注意:** 本番DBへの直接接続は非推奨。必要な場合は `fly proxy` を使用:
```bash
# Fly.io PostgreSQLにローカルからアクセス
fly proxy 15432:5432 --app my-app-db

# 別ターミナルで
DATABASE_URL="postgresql://user:pass@localhost:15432/app" npx prisma studio
```

## シード（初期データ投入）

### package.json設定
```json
{
  "prisma": {
    "seed": "npx tsx prisma/seed.ts"
  }
}
```

### seed.ts例
```typescript
import { PrismaClient } from "@prisma/client";

const prisma = new PrismaClient();

async function main() {
  await prisma.user.upsert({
    where: { email: "admin@example.com" },
    update: {},
    create: {
      email: "admin@example.com",
      name: "Admin",
    },
  });
}

main()
  .then(() => prisma.$disconnect())
  .catch((e) => {
    console.error(e);
    prisma.$disconnect();
    process.exit(1);
  });
```

```bash
# シード実行
npx prisma db seed
```

## トラブルシューティング

### `provider = "sqlite"` のままデプロイしてしまった
```bash
# schema.prisma の provider を "postgresql" に修正
# migration_lock.toml も更新
# マイグレーション再作成
# 再デプロイ
fly deploy
```

### `migration_lock.toml` の provider が不一致
```
Error: The migration_lock.toml provider does not match the schema provider
```
→ `migration_lock.toml` の `provider` を `schema.prisma` と同じ値に修正

### マイグレーション適用エラー（本番）
```bash
# Fly.ioのログでエラー内容を確認
fly logs --app my-app

# DB直接接続でテーブル状態を確認
fly postgres connect --app my-app-db

# SQL直接実行
SELECT * FROM "_prisma_migrations";
```

### Prisma Client生成エラー（ビルド時）
```
Error: @prisma/client did not initialize yet.
```
→ Dockerfileで `RUN npx prisma generate` が `RUN npm run build` の前にあるか確認

### 既存SQLiteデータの移行
- 開発データは移行不要（本番は空DBから開始）
- 既存の本番SQLiteデータがある場合は `pgloader` 等のツールを使用
