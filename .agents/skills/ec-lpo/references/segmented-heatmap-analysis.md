# セグメント比較ヒートマップ分析 詳細ガイド

## 核心原則

```
LP改善は「勘」ではなく「科学」である。

最大の間違い:
├── ❌ 訪問者"全体"のヒートマップを見て判断する
│   ├── 「ここがよく読まれてるな」→ OK!
│   ├── 「ここは飛ばされてるな」→ カットしよう
│   └── 結果: 良かったり悪かったり...正解がわからない
│
└── 全体データはノイズだらけ。本当に見るべきは:
    ├── 🙆 CVしたユーザーの行動 = 「成功の設計図」
    │   └── どの情報に惹きつけられ、どの言葉を信じ、
    │       どのタイミングで購入を決めたか。答えがすべて詰まっている
    └── 🙅 CVしなかったユーザーの行動 = 「失敗の原因リスト」
        └── どこで興味を失い、どこで不安を感じてページを閉じたか。
            改善すべき箇所が丸見えになる
```

## セグメント設定の実務手順

### Microsoft Clarity でのセグメント設定

```
CV者セグメントの作成:
├── 1. Clarity ダッシュボード → フィルター
├── 2. 「Smart events」または「Custom tags」で購入完了イベントを定義
│   ├── 方法A: URLフィルター（サンクスページURL含む）
│   │   └── Page URL → Contains → "/thanks" or "/order-complete"
│   ├── 方法B: JavaScript API でカスタムタグ送信
│   │   └── clarity("set", "conversion", "purchase");
│   └── 方法C: GA4連携でコンバージョンイベント連携
├── 3. フィルター保存 → 名前: 「CV者」
└── 4. 同様に「非CV者」= 上記条件の否定（NOT）で作成

Shopify の場合:
├── サンクスページURL: /checkout/thank-you
├── または Order Status Page URLでフィルター
└── ★ Shopify Plus以外はcheckoutカスタマイズに制限あり → URL方式推奨
```

### Hotjar でのセグメント設定

```
CV者セグメントの作成:
├── 1. Hotjar → Heatmaps → 対象ページ選択
├── 2. 「Compare」機能を使用
├── 3. フィルター条件:
│   ├── Segment A（CV者）:
│   │   └── Event → 購入完了イベント（Hotjar Eventsで事前定義）
│   └── Segment B（非CV者）:
│       └── Event → 購入完了イベント NOT
├── 4. 比較ビューで並べて分析
└── ★ Hotjar Eventsの事前設定が必須
    └── window.hj('event', 'purchase_complete'); をサンクスページに設置
```

### Ptengine でのセグメント設定

```
CV者セグメントの作成:
├── 1. Ptengine → ヒートマップ → セグメント
├── 2. ゴール設定で購入完了を定義
├── 3. セグメント作成:
│   ├── CV者: ゴール達成ユーザー
│   └── 非CV者: ゴール未達成ユーザー
├── 4. ヒートマップ表示でセグメント切替
└── ★ 日本語UIで操作しやすい
```

## 比較分析の具体的な読み方

### 1. スクロールマップ比較

```
スクロールマップの読み方:

CV者のスクロールマップ:
├── 典型的パターン: ファーストビュー(100%) → 中盤(80%) → CTA(70%) → 下部(50%)
├── 注目点: CTAセクションまでのスクロール到達率
└── ★ CV者の多くが到達しているセクション = 購入決定に寄与した情報

非CV者のスクロールマップ:
├── 典型的パターン: ファーストビュー(100%) → 急減(40%) → CTA未到達(15%)
├── 注目点: 「急激にスクロール率が下がるポイント」= 離脱ポイント
└── ★ この離脱ポイントの改善が最優先

比較から読み取れるインサイト例:

パターン1: 非CV者がセクション3で大量離脱
├── 原因仮説: セクション3の内容に興味を持てない / 不安を感じた
├── 対策案A: セクション3の内容をCV者が熟読するセクションに入れ替え
├── 対策案B: セクション3の手前に興味を引くフック（数字/画像）を追加
└── 対策案C: セクション3をカットして短縮

パターン2: CV者と非CV者で到達率に差がないのにCVRが低い
├── 原因仮説: 情報は読んでいるが「購入の決め手」が不足
├── 対策案A: CTAの手前に「今買う理由」を追加
├── 対策案B: CTA自体の訴求力を改善
└── 対策案C: 価格の見せ方・オファーを改善
```

### 2. 熟読（アテンション）マップ比較

```
熟読マップの読み方:

4つのパターンマトリックス:

           │ CV者が熟読する    │ CV者が読まない
───────────┼──────────────────┼──────────────────
非CV者が   │ ★ 共通関心領域    │ 🔴 離脱者の迷い
熟読する   │ →このままでOK     │ →不安/疑問の発生源
───────────┼──────────────────┼──────────────────
非CV者が   │ 🟢 CVの決め手     │ ⚪ 両者無関心
読まない   │ →もっと読ませる!  │ →カット候補
───────────┴──────────────────┴──────────────────

各パターンの対応:

🟢 CV者だけが熟読（CV決定の鍵）:
├── 意味: この情報が購入の決め手になった
├── 行動: この情報をもっと目立つ位置に移動 / 到達率を上げる
├── 例: 「お客様の声」セクション
│   ├── CV者は熟読しているが、非CV者はほとんど読んでいない
│   ├── → 「お客様の声を読ませることができればCVRは上がる」
│   └── → 施策: 序盤に持ってくるA/Bテスト / 手前の訴求を強化
└── ★ 最も改善インパクトが大きい領域

🔴 非CV者だけが熟読（離脱の原因）:
├── 意味: ここで不安/疑問を感じて立ち止まっている
├── 行動: この箇所の内容を改善して不安を解消する
├── 例: 「返品・返金ポリシー」セクション
│   ├── 非CV者が長時間読んでいる = 返品への不安が購入を妨げている
│   └── → 施策: 返金保証の強調 / 「安心してお試しください」のメッセージ追加
└── ★ 不安解消 = CVR向上の直接的な施策

★ 共通関心（両者が熟読）:
├── 意味: 万人が興味を持つ重要コンテンツ
├── 行動: 現状維持。ただし質をさらに高める余地はある
└── 例: 商品の主要機能説明 → このままの位置・内容でOK

⚪ 両者無関心（誰も読まない）:
├── 意味: 不要コンテンツの可能性
├── 行動: カットを検討 / 内容の大幅刷新 / 位置変更
└── 例: 会社概要のような汎用セクション → カットしてLP短縮
```

### 3. クリックマップ比較

```
クリックマップの読み方:

CV者のクリックパターン:
├── CTAボタンへの直線的な動線
├── 画像拡大のクリック（商品への興味）
├── FAQ項目のクリック（最終確認）
└── ★ 「購入に至る合理的な経路」が見える

非CV者のクリックパターン:
├── リンクのない場所への無駄クリック（迷い/不満の証拠）
├── 「戻る」ボタン・ナビゲーションへのクリック（離脱意図）
├── 画像・テキストへの期待クリック（もっと情報がほしい）
└── ★ 「何を探しているのに見つけられないか」が見える

差分から導出される施策:

差分1: 非CV者が「価格」付近を繰り返しクリック
├── 仮説: 価格の詳細/内訳を知りたい
├── 施策: 価格の内訳表示 / 1日あたりコスト表示 / 分割払い表示
└── A/B案: 価格セクションに「コスト比較表」を追加

差分2: 非CV者がロゴ/ナビゲーションをクリック
├── 仮説: LPから離脱して他の情報を探そうとしている
├── 施策: LP内で必要な情報を完結させる
└── A/B案: ナビゲーションの非表示 / LP内アンカーリンクに変更

差分3: 非CV者が画像をクリック（拡大されない）
├── 仮説: もっと商品を近くで見たい
├── 施策: 画像の拡大機能追加 / 詳細画像の追加
└── A/B案: ライトボックス式の画像ギャラリー追加
```

## PC/スマホ別分析の重要性

```
デバイス別分析が必須な理由:

EC事業のモバイル比率: 80-95%（一般的なD2Cブランドの場合）

PC vs スマホ で行動が全く異なる例:
├── PC:
│   ├── スクロール深度が深い（じっくり読む傾向）
│   ├── 比較検討行動が多い（タブ切替で競合比較）
│   └── CVRが相対的に高い場合がある
├── スマホ:
│   ├── スクロール深度が浅い（短気、サムストップ重要）
│   ├── 片手操作 → タップ領域の大きさが重要
│   ├── 読み込み速度の影響が大きい
│   └── ★ ユーザーの大半がスマホ → スマホ最適化が最優先
└── 分析時:
    ├── 必ずPC/スマホを分けてヒートマップを確認
    ├── スマホでのCV/非CV比較を最優先で実施
    └── PC用の改善とスマホ用の改善は別施策として管理
```

## ケーススタディ: よくある発見パターン

```
パターン集（EC LPでよく見る比較結果）:

Case 1: 「お客様の声」問題
├── 発見: CV者はレビューセクションを熟読、非CV者はスキップ
├── 原因: レビューセクションがLP下部にあり、非CV者が到達前に離脱
├── 施策: レビュー抜粋をFV直下に配置 / スクロール途中にも分散配置
└── 結果目安: CVR +15-30%の改善事例あり

Case 2: 「価格ショック」問題
├── 発見: 非CV者が価格セクションで長時間滞留→離脱
├── 原因: 価格を見て「高い」と感じ、その価値が伝わっていない
├── 施策: 価格の手前に「投資対効果」「1日あたりコスト」を配置
└── 結果目安: CVR +10-20%の改善事例あり

Case 3: 「FV離脱」問題
├── 発見: 非CV者のスクロール率がFVの時点で急落（60%以下）
├── 原因: FVで「自分ごと」と感じられない / 広告とのメッセージ不一致
├── 施策: FVのキャッチコピー変更 / ターゲット明示 / 広告との一貫性強化
└── 結果目安: CVR +20-50%の改善事例あり

Case 4: 「CTA見えない」問題
├── 発見: 非CV者のCTAクリック率が極端に低い（CV者の1/5以下）
├── 原因: CTAのデザインが目立たない / コピーが弱い / 配置が少ない
├── 施策: CTAの色・サイズ変更 / コピー改善 / 複数箇所に配置
└── 結果目安: CVR +10-25%の改善事例あり

Case 5: 「モバイル離脱」問題
├── 発見: スマホのCVRがPCの1/3以下
├── 原因: スマホでの文字の読みにくさ / タップ領域の小ささ / 表示速度
├── 施策: SPファーストのデザイン / フォントサイズ拡大 / 画像最適化
└── 結果目安: SP CVR +30-50%の改善事例あり
```

## 分析サイクルの頻度

```
定期分析サイクル:

週次（所要時間: 30分）:
├── CVR数値の確認
├── 異常値の検知
└── 実行中A/Bテストの中間確認

月次（所要時間: 2-3時間）:
├── セグメント比較ヒートマップの詳細分析
├── 新しいインサイトの抽出
├── 次月のA/Bテスト計画
└── 前月テスト結果のレビュー

四半期（所要時間: 半日）:
├── LP全体の構成見直し
├── セグメント定義の見直し（CVの質の変化）
├── 競合LP分析
└── 大規模改善計画の策定

★ 最低月1回のセグメント比較分析を継続することが重要
★ データが蓄積されるほど分析精度が向上する
```
